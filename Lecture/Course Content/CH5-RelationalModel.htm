Introduction

Whenever we design databases we are faced with a number of problems relating to things like data integrity, security, efficiency. We are also faced with problems relating to the structure of the data we are planning to use. For example how do we build a database to store something like an invoice? Although an invoice appears quite straightforward it can contain a number of quite distinctive elements:

This invoice has a number of parts:

The parts are all related because they are all on one invoice which records some business which took place between Frank and Maisie on the one hand and Adrian on the other. 

In this module you will learn how to analyse the structure of something like an invoice and how to design a database from the results of your analysis.

Before we get into the detail though we must clarify some of the terms we use to describe data structures and databases.

Entities

In the real world we view identifiable and informative objects as entities, things with real and individual existence. A human being is an entity, identified by certain properties, characteristics or attributes. In simple terms we could say that the attributes of a human being are such things as name, race, eye colour, hair colour, height etc. Each attribute of the entity has a value.

Tables and table properties

The table structure has some distinct properties

  1. Each entry in the table represents one data item, there are no repeating groups.
  2. The table is column homogenous, i.e. in a given column all items are of the same kind.
  3. Each column is assigned a distinct name. It cannot be the same as any other column in the same table.
  4. Duplicate rows are not allowed, each row must be identifiable by a unique key.
  5. Rows and columns can be viewed in any sequence at any time without affecting the table contents or the semantics of any function using the table.

A flat file does not contain any repeating fields, a table is simply a two-dimensional (tabular) representation of the flat file.

The table data might be stored in a delimited format, for example comma-delimited (cdl) or tab-delimited.
